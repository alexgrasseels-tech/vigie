- (decision) Outil perso mono-utilisateur pour un chef de projet digital gerant ~25 projets/clients distincts
- (decision) Relances auto-adressees a soi-meme (pas au client); l'envoi du mail au client reste manuel dans un premier temps
- (decision) Cockpit = triage du matin: clients a relancer/communiquer aujourd'hui + projets 'en danger' necessitant une action de ma part; PAS de planification/ordonnancement
- (decision) Douleur reelle: oubli de dire au client qu'il devait tester -> projet bloque; veut mettre plus de pression sur les clients et etre plus avenant/proactif dans les communications
- (decision) DEUX mecanismes de rappel: (1) alertes liees a une echeance precise + (2) cadence recurrente 'relance tous les X jours' par client, independante des deadlines
- (decision) Signaux 'projet en danger' retenus: silence trop long (>X jours sans contact), echeance proche/depassee, action client en attente depuis trop longtemps. NON retenu: statut bloque generique
- (decision) Mode de travail choisi: Fast path (redaction directe avec hypotheses taguees)
- (assumption) Saisie manuelle des projets (client, echeance(s), cadence X, statut, 'en attente de quoi') + date du dernier contact maj manuellement pour alimenter le compteur de cadence
- (change) Ajout brique 'conscience de phase': chaque projet a une phase courante; chaque phase porte des actions attendues (organiser tests, livrer, poser un delai). Le cockpit signale l'action de phase oubliee. != planification (pas d'ordonnancement), c'est de la detection d'oubli
- (decision) CONFIRME: le marquage manuel 'j'ai contacte' remet le compteur de cadence a 0 (action declenchante)
- (decision) CONFIRME: fenetres d'alerte d'echeance = J-3 / J-1 / jour J
- (decision) CONFIRME: nom du produit = Vigie
- (decision) CONFIRME: digest e-mail quotidien des relances inclus en V1 (canal = cockpit in-app + digest mail quotidien)
- (decision) CONFIRME: modele de phases par defaut accepte: cadrage -> conception -> developpement -> tests -> livraison -> cloture
